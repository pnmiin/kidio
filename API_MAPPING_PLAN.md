# API Mapping Plan

Frontend: https://kidio-learning.vercel.app/

Backend API base URL: https://kidio-be.onrender.com

Swagger UI inspected: https://kidio-be.onrender.com/swagger/index.html

Working OpenAPI JSON inspected: https://kidio-be.onrender.com/swagger/v1/swagger.json

Do not use `/swagger/index.html` as the API base URL.

## Notes For Implementation Later

1. The Sign Up flow should call only `/api/Auth/register`. It should not call `/api/Auth/login` from the Sign Up button. Because `RegisterResponse` does not contain tokens, the safest UI flow is: register success -> show success -> switch/navigate to Login.
2. The Login flow should call only `/api/Auth/login` and save `AuthResponse` tokens/user.
3. Child profile creation is parent-authenticated through `POST /api/Child`; there is no separate kid credential login endpoint in the inspected Swagger.
4. Game completion can be mapped consistently through `POST /api/Progress/submit`, but most current games have static frontend question/content data. To make content backend-driven, either use `LessonResponse.contentJson` or add dedicated game-content endpoints.
5. Parent-child linking by arbitrary Kid ID is not present in Swagger. Current backend model appears to be parent owns children created via authenticated `POST /api/Child`.
6. Admin analytics are mostly not represented by Swagger endpoints. Existing `adminMockData.ts` cannot be fully replaced without new admin analytics endpoints.
